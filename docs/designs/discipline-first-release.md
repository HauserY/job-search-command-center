## Vision

### 10x Check
The app doesn't just track your discipline, it runs your campaign: you pick a plan, it sets your targets, tells you each week whether your current pace mathematically reaches an offer, and coaches the gap — via local/BYO-key LLM, still private. That full version (Campaigns & Playbooks + copilot) stays out of scope per D11. This release ships the weekly retro (the first genuinely copilot-shaped feature — the app reflects back instead of recording) plus the PWA (daily-retention infrastructure that keeps the habit loop alive; not copilot-related, but launch-critical for a daily tool).

Positioning: JobSync wins "AI job tracker"; JSCC wins **"private campaign discipline system."** The headline is the discipline engine — streaks, daily targets, recruiter cadence — which no other OSS tracker has. Local-first/no-account is the trust line, not the headline.

Replaces the current catch-all in src/state/store.jsx:7-16 whose fresh-state fallback gets persisted over the user's data by the debounced save. When stored `schemaVersion` is newer than the build (skew guard), show a visible "newer version detected — refresh to update" message rather than failing silently.

- **Cross-context state sync (finding 4B):** listen for the native `storage` event on `jscc_v1`; when another tab/installed-PWA context writes, re-hydrate this context's state from disk (defer while an input has focus). Eliminates silent last-writer-wins loss between an installed PWA and a browser tab. Known documented limit: simultaneous edits in both contexts within the debounce window.

Deep audit deferred to post-implementation /design-review.
- **Top-level error boundary (finding 8A):** catches any render crash; fallback says "Something went wrong — your data is safe," offers Export + reload, and shows expandable copyable error details so bug reports arrive with stack traces. The render-time counterpart to the load-time quarantine screen. README states "no analytics, no tracking" as an explicit feature.
- **Persist-failure banner (finding 2B):** save() catch (currently empty, src/state/store.jsx:27) sets a `persistFailed` flag; a persistent banner shows "Your changes aren't being saved — export your data now" with the export action inline; clears on next successful save. Covers quota-exceeded and storage-disabled browsers.

Cherry-picked additions (this ceremony):
- **Weekly retro view** — "Last week" card on Insights. Spec: a week is the existing `weekKey()` convention (dates.js); "last week" = the most recent complete week. Metrics: counter totals per task summed over the week's dailyHistory entries, compared against **snapshotted** targets (decision T3, outside voice accepted): rollover snapshots that day's targets into the dailyHistory entry; each week is judged against its own targets; migration defaults pre-existing entries to current targets, labeled as approximated. Weekly target = daily target × 7. Outreach activity shown as **two honest counts** (decision T4): "outreach sent: N" and "replies received: M" for the week — no same-week ratio (delayed replies make it structurally wrong; cohort conversion math ships with the deferred forecast feature). Bucketing by `weekKey(createdAt.slice(0,10))` matching the existing weeklyTrends convention in src/lib/insights.js. Top source = source with most new opportunities created that week (bucketed the same way on opportunity `createdAt`). Streak status from existing computeStreaks logic (currently a private function in src/tabs/Today.jsx:47 — extract to src/lib/ for shared use rather than duplicating). Edge cases tested: sparse/empty weeks, first-ever week (no complete week yet), week boundaries, targets changed mid-week.
- **Installable PWA** — vite-plugin-pwa 1.3.0 (Vite 8 compatible, verified): manifest, icons, offline precache of the app shell. Update strategy named explicitly: `registerType: 'autoUpdate'` with the default workbox `generateSW` precache; because state is versioned via the migration registry (sequenced first), a stale cached build writing old-schema state is migrated safely on next load. Reverse-skew safety is also required: app code must spread-preserve unknown state fields on load/save (never strip fields it doesn't know), and must refuse to persist when the stored `schemaVersion` exceeds the version the running build knows — checked **at write time** (re-read localStorage inside the debounced save), not only at load, since a stale tab loaded before the newer tab wrote would pass a load-time check and still clobber. Manifest `scope`/`start_url` set for the Pages subpath.

## Spec Clarifications (outside voice, adopted at T6 — 2026-07-17)

1. **No validation contradiction:** import-boundary strictness (reject unknown top-level keys in backup FILES; refuse newer-version payloads outright) and runtime field-preservation (spread-preserve unknown fields in LOCALSTORAGE state under version skew) are different layers with different trust models. Both stand.
2. **Second demo exit — "keep this data":** the demo banner offers BOTH "keep this data" (clears `demoMode`, keeps state, banner disappears — for users who turned demo data real) AND "clear demo data and start fresh" (confirm dialog per finding 4A). Fixes the seeded-once vs still-demo conflation.
3. **Onboarding flag transition table:**
   | Entry path | `onboarded` | `demoMode` |
   |---|---|---|
   | First visit (no key) | false → onboarding shows | false |
   | "Start with demo data" | true | true |
   | "Start empty" | true | false |
   | Import backup (any screen) | true | from payload |
   | Recovery → start fresh | true | false |
   | Recovery → import | true | from payload |
   | Demo banner → keep data | unchanged (true) | false |
   | Demo banner → clear | true | false |
   | Migration of pre-release state | true (existing user) | false |
4. **Cross-tab sync limits documented:** storage-event re-hydration prevents silent overwrite in sequential use; simultaneous edits in two contexts within the debounce window are last-writer-wins and are NOT merged. Stated in README/docs.
5. **Launch assumption stated:** this is the first public release — no stale installed builds exist in the wild; the skew guards protect all FUTURE deploys.
6. **README privacy phrasing:** "the app contains no analytics or tracking code" (GitHub Pages infrastructure logs are outside the app's control).
7. **Env-driven base:** Vite `base` (and PWA `scope`/`start_url`) derive from an env var/config with a sensible default, so forks and local previews don't break on the hardcoded repo name.
8. **Playwright serving spec:** the smoke test serves the production `dist/` via a static server mounted at `/job-search-command-center/` on localhost (SW registration is permitted on localhost without HTTPS); the test also asserts the lazy Insights chunk is present in the SW precache manifest.
9. **60-second criterion as acceptance test:** from cold load of the Pages URL on a throttled "Fast 3G" profile: first paint < 5s, "Start with demo data" visible without scrolling, one click seeds demo, Today renders populated (checklist + rings + streak + heatmap) — total elapsed < 60s. Pre-push checklist itemized: current-tree file audit (fixtures, defaults, docs), README image/GIF content + metadata, no localStorage dumps or generated artifacts committed, fresh-history verification, LICENSE/name/base consistency. README copy note: position as "private campaign discipline system"; do not frame against "AI job tracker" in a way that reads as AI-avoidance.

### Engineering Review Addenda II (outside voice, adopted at finding 8 — 2026-07-18)

Concurrency decision (tension 7, review position kept): last-writer-wins within the sub-second debounce window stays a DOCUMENTED limit; focus-defer + hydrate-storm guard are the shipped mitigations; revision-based conflict detection is a TODOS.md item gated on real-world reports.

1. Error boundary's Export action reads raw `localStorage` directly — never React state — so it works even when the state layer is what crashed.
2. Quarantine key is timestamped (`jscc_v1_corrupt_<ISO>`) and retained until the user explicitly deletes it; the recovery screen can re-offer the download any number of times.
3. Import size cap: **10 MB**, error message names the limit ("Backup exceeds the 10 MB import limit").
4. Demo seed idempotency enforced in the reducer via a `demoSeeded` guard flag — not via disabled-button UI.
5. PWA icons specified: 192px + 512px plus maskable variants, generated with `@vite-pwa/assets-generator` (already a peer of vite-plugin-pwa).
6. Smoke test also covers reload-after-SW-update and offline tab navigation including the lazy Insights chunk; cold-offline-first-visit is explicitly out of scope and documented. First-visit network cost vs post-install precache cost are distinct metrics — the lazy split optimizes the former; precache still downloads the chunk in the background after install (by design).
7. 60-second acceptance test parameters pinned: mid-tier laptop, Chrome stable, cold cache, 1366×768, DevTools throttling Fast 3G + 4× CPU.
